Managing Clusters

by ipedrazas 2015-03-04 Cassandra

There are plenty of solutions out there that try to solve this problem. In fact, depending on your problem, you should chose one solution or another.

For example, Apache Mesos that abstracts machine resources like CPU, memory or storage, or Yarn (from the Hadoop family) that allows you to schedule your MapReduce jobs.

At Adbrain I’ve built a little tool that does a bit of both. By any means it’s anything like Mesos or Yarn but a super-hyper striped out version of the core ideas behind these two wonderful systems.

I’ve called it “CheckPoint” because I’ve designed it following the Hollywood Principle: “Don’t call me, I’ll call you“. One of the most annoying things I’ve found when provisioning machines using code is all that polling. Don’t get me wrong, it works, but if Javascript has showed us something is that callbacks are (awesome) way better to deal with than threads and sleeps (or waits). CheckPoint works the other way around. For example, it creates and provisions the Apache Spark master, once it’s up and running, that machine sends a request to Checkpoint and it creates and provision the workers. The whole cluster creation is asyncronous and it decoupled of what the other resources do.

So, CheckPoint started like yet-another-provisioning tool, but it ended up being much more than that. CheckPoint was the tool that allowed us to start working differently in what I call “Agile Clusters“.

The foundation of Agile Clusters are 3 simple elements:

  1. Creation and provisioning of Ephemeral clusters.
  2. Ephemeral logging and monitoring.
  3. Destruction of idle resources.

The 1st point it’s easy, you need a way of building your machines. There are many solutions like Puppet, Ansible, Salt or Chef. CheckPoint uses Chef because that’s what we use at Adbrain, but it’s more of a burden than an advantage.

The 2nd point is quiet tricky. Ephemeral infrastructure is great, but when the machine go away, so goes the data, and the logs, and everything you didn’t take out first. The other itchy point is monitoring. Yes, there are plenty solutions are there, but not that many that can deal with ephemeral infrastructure. CheckPoint uses InfluxDB and grafana, in particular the “Scripted Dashboards”

The 3rd point is a blessing in disguise. CheckPoint provides a REST API so, developers can see infrastructure as one step more in their workflow execution where they can create and destroy resources but if for any reason your application or a developer forgets about a cluster… CheckPoint does not an it will destroy that resource unless it’s used.

There’s something that has made a huge impact in how we work: Accessible Information.

I’ve been working in Knowledge Management almost all my life and I know how important is to have access to the right data at the right time.

That’s why when I was building CheckPoint I was very conscious of who was going to use it and what it might be helpful to know at that time.

For example, the fact of displaying CPU, RAM and Price when selecting your instances has been incredibly useful to improve our cluster efficiency and cost.

But we not only managed to reduce the infrastructure cost by 30%, we achieved something that until now was a big unknown: “How much it costs to run every big data job”, because neither Mesos nor Yarn worry about cost, their job is not juggling with cost, it’s solving another problem, quite a hard one, but to me being able to quantify how much it cost to run a job is critical, because it’s not about in this world of Cloud services it’s not about if it’s possible or not, it’s about if economically makes sense or not.
